# Database changes — how we track them

This Mac can't run the Docker-based `supabase db push`, and the MCP connection
is read-only, so DB changes are applied **by hand in the Supabase dashboard
SQL Editor**. To keep the schema history in git anyway, every change lives here
as a numbered file.

## Workflow (every time the database changes)
1. The SQL is written here first as `NNNN_short_description.sql` (next number).
2. You paste it into Supabase → **SQL Editor** → Run.
3. It's committed to the repo. The folder = the chronological record of every
   change since the baseline.

## Conventions
- **Number** sequentially: `0001_`, `0002_`, … (zero-padded).
- **Idempotent** where possible: `IF NOT EXISTS`, `DROP … IF EXISTS` before
  `CREATE`, `ON CONFLICT DO NOTHING`, `ADD COLUMN IF NOT EXISTS`. So re-running
  a file is safe.
- One logical change per file; a short comment at top says what & why.
- Big one-time data seeds live in `../seeds/` (not numbered) and are noted in
  the migration that needs them.

## Baseline
`../schema_snapshot.sql` is the authoritative snapshot of the schema as it was
in April 2026 (generated by read-only introspection). Migrations here are the
changes applied **after** that snapshot.
